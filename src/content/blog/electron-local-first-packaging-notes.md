本地优先的桌面应用和普通 Web 应用最大的不同，是它必须认真处理文件系统、窗口生命周期、预加载桥接和打包分发。Electron 让这些能力变得可用，但边界设计仍然需要谨慎。

noteDock 的工程结构分为 main、preload 和 renderer。主进程负责桌面能力，预加载脚本负责暴露安全的桥接 API，渲染进程负责 React UI 和编辑体验。

## 主进程边界

主进程适合处理窗口、菜单、文件对话框、路径读写和系统事件。它不应该承载过多业务 UI 状态，否则调试和复用都会变得困难。

对 noteDock 这类文档工具来说，主进程最核心的职责是提供稳定的本地文件能力：打开目录、读取文件、保存内容、监听工作区变化，以及在需要时调用系统能力。

## preload 的价值

preload 是桌面能力和 Web UI 之间的边界。它应该暴露清晰、有限、语义化的 API，而不是把 Node 能力直接交给页面。

例如渲染进程不需要知道具体的 `fs` 调用细节，它只需要 `openDirectory`、`readDocument`、`saveMarkdown` 这类面向业务的接口。这样 UI 层更接近普通前端，安全边界也更清楚。

## 渲染层体验

渲染层的复杂度主要来自编辑器、文件树、多格式预览和弹窗工具。React 很适合组织这些交互，但需要避免把所有状态放进一个巨大的组件里。

比较稳妥的做法是让文件工作区、编辑器状态、预览器状态和工具弹窗各自保持独立，再通过明确事件串联。这样当后续加入 PDF、Word、Excel、Excalidraw 或 React Flow 时，不会互相挤压。

## 打包与分发

桌面应用最终要走到构建和安装包。electron-builder、图标资源、Windows 安装器素材、构建脚本和运行时检查都属于产品体验的一部分。

本地优先应用的优势是数据掌握在用户自己手里。工程上需要做的，是把这种优势变成稳定、可信、可安装、可长期使用的桌面体验。
