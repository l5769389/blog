本地 Markdown 工具的第一层价值是写作，第二层价值是组织。只有当文档之间的关系能被解析和呈现时，文件夹才会逐渐变成知识库。

noteDock 中的知识模型围绕几类信息展开：文档标题、frontmatter、标签、wiki link、普通链接、反向链接和文件更新时间。它们不需要一开始就形成复杂图谱，但需要被稳定地抽取和更新。

## 为什么从轻模型开始

个人知识库很容易被“图谱感”吸引，但真正影响使用频率的是基础信息是否可靠。标签能不能被识别，链接能不能跳转，重命名后引用是否还能定位，这些细节决定了工具是不是可信。

轻模型的好处是可维护。每个 Markdown 文件仍然是普通文本，迁移成本低；额外的关系信息来自解析层，不会把用户锁进某个特殊格式。

## wiki link 与反向链接

`[[文档标题]]` 这类 wiki link 适合表达“这篇笔记和另一篇笔记有关”。当解析器能收集这些链接，再反向生成“哪些文档引用了我”，文档之间就有了双向导航。

反向链接不一定需要复杂 UI。一个清晰的引用列表、来源片段和跳转入口，已经能帮助用户在项目复盘、技术调研和长期笔记中找到上下文。

## frontmatter 与标签

frontmatter 更适合放结构化元数据，例如状态、项目名、日期、来源和分类。标签则更适合表达跨目录的主题，比如 `Electron`、`DICOM`、`Signal Processing`。

这两者可以互补：frontmatter 负责明确字段，标签负责轻量聚合。对本地工具来说，重点是保持格式简单，让用户不用学习复杂规则也能获得组织能力。

## 搜索与导航

当知识模型稳定后，搜索就不只是全文匹配。它可以结合标题、标签、链接关系、更新时间和当前工作区位置，给用户更贴近语境的结果。

一个好的本地知识工具应该让用户知道“我在哪个文件里、这个文件属于哪个项目、它和哪些内容有关”。noteDock 的知识关系模型就是在为这种导航体验打基础。
